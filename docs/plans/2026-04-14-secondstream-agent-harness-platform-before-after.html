Agent Harness v1 — Vista de plataforma completa (no solo workspace)

Este artefacto explica el salto entre la plataforma actual y la plataforma con Agent Harness v1. El workspace es la primera superficie visible, pero el harness es platform-wide: captura, evidencia, contexto, memoria, sesiones/runs, supervisión e inbox cross-stream.

Tesis visual: con Harness v1, SecondStream no solo guarda campos o sugerencias; empieza a operar un sistema donde agentes y humanos son usuarios operativos sobre la misma plataforma, con truth estructurada + memoria portable + artifacts auditables + supervisión explícita, sin depender de memoria stateful propietaria del provider.

Plataforma hoy (frame dominante)

Workspace-first AI como helper parcial
  • Las capacidades existen (intake, voice, files, workspace, review), pero están conectadas como features, no como harness explícito.
  • El broker consolida entendimiento del caso principalmente en su cabeza.
  • No hay objeto central de comprensión versionada del stream como artefacto operativo persistente.
  • No hay modelo de session/run como contrato visible de ejecución agente.
  • No hay inbox cross-stream para supervisión sistemática del trabajo de agentes.

Dónde se siente la fricción

Contexto fragmentado Memoria no compuesta
  • Las sugerencias AI viven por campo/flujo, pero no hay “current working understanding” duradera.
  • El estado de razonamiento no se inspecciona como run history + artifacts + feedback loop.
  • El sistema no trata todavía al agente como usuario operativo de primer nivel.
  • La mejora incremental de agent behavior carece de control-plane explícito.

Plataforma con Agent Harness v1 (frame objetivo)

Platform-wide harness Workspace = primera superficie visible Model-agnostic SecondStream owns memory/context/artifacts

Capture / Intake

Voice, files, notes, quick capture como entradas gobernadas.

Discovery / Workspace

Discovery Brief + Workspace Suggestions + revisión broker.

Session / Run

Ejecuciones materiales trazables, con triggers y estados.

Memory / Knowledge

Observations, dossiers históricos, promoción controlada a org knowledge.

Supervisión

Agent inbox, run history, pending reviews, fallos, aprobaciones.

Team Outputs

Superficies futuras de management y query operacional grounded.

Diff de plataforma: de app con AI a harness productivo

Antes

  • Centro operativo: formulario/workspace + suggestions.
  • AI ayuda en extracción, no opera como actor gobernado integral.
  • Contexto ensamblado de manera implícita por flujo.
  • Sin session/run como contrato operativo principal.
  • Memoria útil dispersa y poco portátil como activo estratégico.
→

Después (Harness v1)

  • Centro operativo: harness de plataforma + workspace como superficie inicial.
  • Agente como usuario operativo con identidad, permisos, scope y policy.
  • Context assembly es responsabilidad explícita de SecondStream.
  • Session/run + artifacts + feedback construyen mejora acumulativa.
  • Memoria y artifacts son activos portables, inspectables y exportables.

Arquitectura de plataforma por capas/superficies

Layer 1 — System of Record (Truth)

Postgres + APIs canónicas + RBAC + estados transaccionales oficiales.

Layer 2 — Agent Control Plane

Identidad de agente, tool registry, policy/approval, lifecycle session/run, observabilidad.

Layer 3 — Memory/Artifact Fabric

Filesystem/Object storage para memoria durable y artifacts; portable, inspeccionable.

Layer 4 — Agent Workforce

Discovery agent primero; luego especialistas (pricing, compliance, follow-up, etc.).

Layer 5 — Human Surfaces

Workspace, review points, inbox/historial, aprobaciones y gobernanza de conocimiento.

Principio transversal

LLM = motor reemplazable. Harness/state/memory = propiedad de SecondStream.

Memoria, contexto, session/run, knowledge, inbox

  • Session: contexto vivo mientras el problema material del stream no cambie.
  • Run: ejecución concreta por trigger (new source, refresh, correction, etc.).
  • Context assembly: retrieval curado de observations, dossiers, knowledge aprobada, evidencias.
  • Knowledge promotion: stream learnings → org knowledge por confirmación múltiple o aprobación humana.
  • Inbox: supervisión cross-stream (briefs pendientes, run failures, promotions pendientes).

Flujo de persistencia (v1)

1) Ingreso de evidencia

Documento/voz/texto ingresa por superficies de captura.

2) Run ejecuta

Agente consume contexto y genera brief/suggestions/artifacts.

3) Broker supervisa

Corrige puntos, aprueba/defer/reject recomendaciones.

4) Memoria durable

Observations + session summaries + artifacts quedan persistidos.

5) Reuso inteligente

Contexto futuro se enriquece sin depender de memoria stateful del proveedor.

Agente como usuario operativo de la plataforma

No es un widget adjunto: es un actor con identidad, permisos, alcance, auditoría y límites de policy.

Capacidad Humano Agente Regla v1
Leer contexto operativo Sí Sí Paridad deliberada de lectura, con scopes y RBAC.
Escribir memoria/artifacts/drafts Sí Sí Permitido sin approval para trabajo derivado no-transaccional.
Cambiar truth canónica Sí No directo Agente propone; humano revisa/acepta.

Truth vs Memory vs Artifacts vs Supervision

Truth (estructurada, transaccional)

Estados canónicos del producto: entidades, lifecycle, aprobaciones oficiales, sync states.

System of record
Memory (durable, portable)

Summaries, observations, dossier links, aprendizaje contextual curado por sesión/stream/org.

Owned by SecondStream
Artifacts (trabajo operativo)

Discovery Briefs, context packs, research outputs, drafts, evidence derivatives, diff/version history.

Inspectable / exportable
Supervision (gobernanza)

Inbox, run history, approvals, fallas, correcciones repetidas, evaluación y mejora continua.

Human-in-the-loop
Regla clave de arquitectura: filesystem/object storage funciona como capa portable de memory/artifacts. No reemplaza toda la verdad estructurada del negocio. Truth y memory son capas distintas y complementarias.
